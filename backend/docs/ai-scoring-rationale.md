# AI를 채점기로 "그냥" 쓰지 않았다 — 설계 근거

> 심사 Q&A 대비용. "그냥 AI한테 점수 매겨달라 한 거 아니냐"는 질문이 들어오면 이 문서를 펼쳐 답한다.
> 모든 수치는 코드와 1:1 — `analysis_pipeline.py` · `gemini_verify.py` · `speech_metrics.py`.

## 한 줄 요약

AI(Gemini)는 **3개 채점 축 중 "내용" 1축에만** 쓰고, 그것도 자유 출력이 아니라 **JSON 스키마로 강제**한다.
전달력·안정성은 **LLM 없이 규칙 기반으로 결정론적 계산**(동일 입력 → 동일 점수). 최종 총점은 세 축의 **가중 합성 + 후함 보정(calibration)** 결과다.

---

## Q1. 그냥 Gemini한테 점수 매겨달라고 한 거 아니냐?

아니다. 채점은 3축으로 분리돼 있고, **LLM이 쓰이는 건 content 한 축뿐**이다.

| 축 | 방식 | LLM? | 재현성 |
| --- | --- | --- | --- |
| **content (내용)** | Gemini가 0~100 채점 | ✅ (스키마 강제) | 비결정론 → 0~100 클램프로 범위 보정 |
| **delivery (전달력)** | WPM·필러 개수 규칙 감점 | ❌ | 결정론(동일 입력=동일 점수) |
| **stability (안정성)** | 침묵·발화비율 규칙 감점 | ❌ | 결정론 |

총점 = `content×0.6 + delivery×0.2 + stability×0.2` (`analysis_pipeline.py`).
즉 점수의 40%는 LLM이 손도 안 댄, 단어 타임스탬프 기반 객관 지표다.

---

## Q2. 왜 STT를 두 개 썼나?

한 엔진으로는 "정확한 정본"과 "필러·타임스탬프"를 동시에 못 얻기 때문이다. 그래서 **이중 STT 교차검증**을 한다.

- **CLOVA** — 정본(맞춤법·문장 정확도) 용도.
- **Whisper** — 필러("음/어")와 단어별 타임스탬프 보존 용도(`vad_filter=False`로 침묵까지 보존).
- 두 전사를 `ThreadPoolExecutor`로 **병렬** 실행하고, Gemini가 둘을 대조해 가장 정확한 `verifiedTranscript`를 만든다.
- 한쪽 실패는 `degraded` 플래그로 표시하고 나머지로 진행 → 단일 장애점이 아니다.

## Q3. 가중치 0.6 / 0.2 / 0.2는 왜 그 숫자냐?

**content가 변별력이 가장 좋아서** 비중을 키웠고, 초기 채점이 "유창하기만 하면 총점이 후하게" 나오던 문제를 **calibration으로 보정**했다.

보정 내역(`analysis_pipeline.py`의 calibration 노브):

- **가중치**: content `0.5 → 0.6`, delivery `0.25 → 0.2`, stability `0.25 → 0.2` — 변별 잘 되는 content로 무게 이동.
- **baseline**: delivery/stability 시작점 `100 → 88` — 유창함만으로 만점이 떠받쳐지지 않게.
- **감점 강화**: 필러 1개당 `4 → 5`점 / 침묵 횟수 계수 `5 → 7`·임계 `2 → 1.5`초 / 발화비율 목표 `0.8 → 0.85`·계수 `100 → 120`.

**효과 (아키타입 시뮬레이션, 총점):**

| 아키타입 | 보정 전 | 보정 후 |
| --- | --- | --- |
| 약한 내용 + 유창한 발화 | 67 | **56** |
| 중간 내용 + 보통 발화 | 73 | **63** |
| 우수 내용 + 유창한 발화 | 94 | **89** |

"말은 매끄럽지만 내용이 부실한" 답변의 점수가 가장 많이 떨어진다(67→56). 즉 가중치는 임의값이 아니라 **변별 목표를 두고 시뮬레이션으로 맞춘 값**이다.

## Q4. 왜 파인튜닝을 안 했나?

**파인튜닝은 안 했고, 그게 이 규모에서 맞는 선택이라고 본다.** (없는 걸 있다고 하지 않는다.)

- 파인튜닝하려면 **사람이 라벨링한 "정답 점수" 데이터셋**이 필요한데, 그게 없다. 라벨 없이 튜닝하면 오히려 편향만 학습한다.
- 대신 **세 가지 더 가벼운 도구**로 같은 목표(일관성·변별력)를 달성했다:
  1. **구조화 출력 강제** — Gemini를 `response_schema`(JSON)에 묶어 출력 형식/필드를 보장(`gemini_verify.py`).
  2. **규칙 기반 객관 지표** — 전달력/안정성은 LLM을 아예 안 쓰고 결정론적으로 계산.
  3. **calibration 노브** — 가중치·감점 계수를 코드 상수로 빼서 데이터 없이도 튜닝 가능(위 Q3).
- 모델은 `gemini-3.1-flash-lite` — 질문 생성·채점 모두 통일. 채점 품질 대비 비용/지연이 적정한 도구 선택.

요지: **"파인튜닝 안 함 = 대충 씀"이 아니라, 라벨 데이터가 없는 상황에서 스키마+룰+calibration이 더 옳은 도구**였다는 것.

## Q5. 점수 일관성·안정성은 어떻게 보장하나?

- **규칙 기반 두 축은 완전 결정론적** — 같은 오디오면 항상 같은 delivery/stability.
- **LLM 축은** ① JSON 스키마 강제로 형식 보장, ② `0~100` 클램프로 범위 보정, ③ `verifiedTranscript` 비면 원본 전사로 폴백.
- **키 없거나 호출 실패해도** 중립 점수(`FALLBACK_SCORE=70`)와 안내 문구로 폴백 → 데모가 안 깨진다.

---

## Q6. 알려진 한계 (먼저 밝힌다)

방어만 하지 않고 한계도 안다는 걸 보여주는 게 더 강하다.

- **무발화/무음 답변의 폴백 점수 과대** — 무음 녹음 시 delivery≈48, stability=0, transcript="". content는 **라이브 채점이면 ~0–10(총점 ~13)** 으로 잘 깎이지만, **폴백 경로에선 `FALLBACK_SCORE=70`이 그대로 적용돼 총점 ~52**가 나온다.
  - *보완 방향*: 무발화 감지 시 폴백 content를 0으로 강등.
- **잡 스토어 휘발성** — 결과를 메모리 dict에 보관 → 프로세스 재시작 시 소멸(데모 범위, DB 없음).
- **CLOVA webm 수용 / Whisper 침묵 검출** — 실제 오디오로 런타임 실측 권장(실패 시 `degraded` 처리됨).

### 한 문장 방어 멘트

> "AI는 내용 채점 한 축에만, 그것도 JSON 스키마로 묶어 썼습니다. 전달력·안정성은 LLM 없이 규칙 기반으로 결정론적으로 계산하고, 총점은 세 축을 가중 합성한 뒤 아키타입 시뮬레이션으로 후함을 보정한 값입니다. 파인튜닝은 라벨 데이터가 없어 안 했고, 대신 스키마 강제·규칙 지표·calibration으로 일관성과 변별력을 확보했습니다."
