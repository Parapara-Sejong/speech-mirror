### 📌 작업 개요
홈 화면을 추가하고 `/` 라우트를 서비스 진입점으로 재편. 기존 폴링 확인 화면은 `/dev`로 분리

이력서·자기소개서 입력부터 직종/면접 종류/인재상 설정, 질문 선택, 마이크 녹음 기반 면접 진행까지 이어지는 프론트 온보딩 흐름 구현

**보고서 파일**: `.report/20260624_#10_홈_온보딩_질문선택_면접흐름_구성.md`

### 🎯 구현 목표
- 사용자가 홈에서 모의면접을 시작해 업로드, 설정, 질문 선택, 면접 진행 화면까지 이동 가능하도록 구성
- 백엔드 질문 생성 API 연동 전에도 mock 질문으로 전체 플로우 확인 가능하도록 처리
- 추후 FastAPI/GEMINI 연동 시 교체할 질문 생성 입력 계약과 multipart 업로드 지점 마련

### 🔧 주요 변경사항 상세

#### 질문 생성 연동 경계
`questionApi.ts`가 질문 생성 호출의 단일 진입점 역할을 하도록 구성. 현재는 400ms 지연 후 `mockQuestions`를 반환하며, 백엔드 구현 후 같은 함수 내부에서 `apiClient.post('/questions/recommend', form)` 방식으로 교체 가능

**특이사항**:
- GEMINI 호출은 프론트에서 직접 수행하지 않음
- PDF/DOCX 텍스트 추출도 프론트 라이브러리를 추가하지 않고 FastAPI 책임으로 남김

#### 파일 입력 처리
TXT/MD는 브라우저 `File.text()`로 미리보기 텍스트를 채우고, PDF/DOCX는 원본 파일만 store에 저장. 사용자가 직접 textarea에 붙여넣는 fallback도 유지

#### 딥링크 fallback
`/questions`, `/interview`로 직접 접근해도 mock 질문을 사용해 화면이 깨지지 않도록 fallback 경로 유지

### 📌 참고사항
- 현재 질문 추천은 mock 데이터 기반. 실제 개인화 질문은 FastAPI `POST /questions/recommend` 구현 후 `questionApi.ts`에서 교체
- `/result`는 별도 mock 리포트 화면을 사용 중이며, 이번 온보딩/면접 흐름의 녹음 결과와 실제 분석 API는 아직 연결되지 않음
